# Plan 문서 운영 규칙

## 목적

코드를 작성하기 전에 목표, 범위, 설계와 검증 기준을 합의하고 그 이력을 저장소에 남긴다. Plan은 구현을 대신하는 문서가 아니라 승인된 구현 계약이다.

## 적용 범위

다음 변경에는 작업 규모와 관계없이 Plan이 필요하다.

- 프론트엔드 또는 백엔드 소스 코드
- 빌드, 린트, 포맷, TypeScript, Python, 환경 예시 등의 설정
- 자동 테스트와 테스트 설정
- 패키지 또는 Python 의존성
- 마이그레이션, 스캐폴딩, 코드 생성 결과
- 실행 동작이나 공개 인터페이스에 영향을 주는 스크립트

다음 변경만 수행할 때는 Plan을 생략할 수 있다.

- Markdown 문서
- 코드 동작을 바꾸지 않는 주석과 오탈자

문서 변경과 코드 변경이 한 작업에 함께 포함되면 Plan이 필요하다. Plan 필요 여부가 불분명하면 작성하는 쪽을 기본값으로 삼는다.

## 파일 위치와 이름

- 모든 Plan은 `plans/`에 저장한다.
- `plans/TEMPLATE.md`를 복사해 작성한다.
- 파일명은 `NNN-english-kebab-case-title.md` 형식을 사용한다.
- `NNN`은 기존 Plan 중 가장 큰 번호의 다음 번호이며 세 자리로 맞춘다.
- `TEMPLATE.md`는 번호 계산에서 제외한다.
- 예: `plans/001-project-setup.md`, `plans/002-todo-list-api.md`
- 완료된 Plan도 이동하거나 삭제하지 않고 `plans/`에 보존한다.
- 코드 구현을 시작하면 Plan과 동일한 번호·제목의 하네스 로그를 생성한다.
- 예: `plans/001-project-setup.md`의 대응 로그는 `logs/001-project-setup.md`다.
- 하네스 로그 작성 규칙은 [HARNESS_LOGGING.md](HARNESS_LOGGING.md)를 따른다.

## 필수 구성

Plan 상단에는 다음 메타데이터를 둔다.

- 번호
- 상태
- 생성일
- 승인일
- 완료일
- 대응 로그 경로

본문에는 다음 여섯 섹션을 반드시 포함한다.

1. 목표
2. 범위
3. 구현 계획
4. API·타입 변경
5. 테스트 계획
6. 완료 기준

API 또는 타입 변경이 없는 작업도 해당 섹션을 삭제하지 않고 `변경 없음`이라고 기록한다.

## 상태와 승인 절차

상태는 아래 순서로 관리한다.

```text
Draft → Approved → In Progress → Completed
```

### Draft

- 조사 결과를 바탕으로 Plan을 작성하거나 수정하는 상태다.
- 읽기, 검색, 분석, 비수정 검증만 허용한다.
- 코드, 설정, 테스트, 의존성, 생성 결과를 변경하지 않는다.
- Plan과 관련 문서 자체는 작성하거나 수정할 수 있다.

### Approved

- 사용자가 채팅에서 해당 Plan의 구현을 명시적으로 승인한 상태다.
- “구현해줘”, “진행해줘”, “이 Plan대로 작업해줘”처럼 구현 의사가 분명한 요청을 승인으로 본다.
- 단순 질문, 검토 요청, 모호한 긍정 표현은 승인으로 간주하지 않는다.
- 승인 즉시 상태와 승인일을 기록한다.

### In Progress

- 승인된 Plan을 기준으로 실제 파일 변경을 시작한 상태다.
- 첫 코드 변경 전에 상태를 `In Progress`로 갱신하고 대응 하네스 로그를 생성한다.
- 승인된 범위 안에서 구현하고 진행 체크리스트를 갱신한다.
- 중요 결정, 의미 있는 실패, 검증 결과가 발생하면 하네스 로그를 수시로 갱신한다.

### Completed

- Plan의 구현과 필수 검증이 끝난 상태다.
- 완료 기준 체크리스트, 실제 검증 명령과 결과를 기록한다.
- 하네스 로그의 작업 요약, 변경 파일, 검증 결과와 미해결 사항을 확정한다.
- 상태와 완료일을 갱신한다.
- 실행하지 못한 검증이 있으면 이유와 영향을 기록한다.
- 필수 완료 기준을 충족하지 못했거나 하네스 로그가 완성되지 않았다면 `Completed`로 변경하지 않는다.

## Plan 변경과 재승인

구현 중 다음 항목이 달라지면 작업을 멈추고 Plan을 수정한다.

- 작업 범위
- 공개 API 또는 라우트
- 데이터 모델이나 타입 계약
- 데이터 흐름 또는 아키텍처
- 프로덕션 또는 개발 의존성
- 테스트 전략 또는 완료 기준

이 경우 상태를 `Draft`로 되돌리고 승인일을 비운 뒤 사용자에게 재승인을 요청한다. 재승인 전에는 코드 변경을 계속하지 않는다.

승인된 범위 안의 파일명 조정, 함수 분리, 내부 변수명, 사소한 스타일 조정처럼 외부 동작과 계약을 바꾸지 않는 구현 세부사항은 재승인 없이 진행할 수 있다. 필요한 경우 Plan의 진행 기록에 남긴다.

## 완료 후 관리

- 완료된 Plan은 작업 이력으로 영구 보존한다.
- 완료된 하네스 로그도 Plan과 함께 영구 보존한다.
- 구현 결과가 Plan과 일치하는지 최종 확인한다.
- Plan, 하네스 로그, 실제 변경 파일과 검증 결과가 서로 일치하는지 확인한다.
- 관련 아키텍처, API, 프론트엔드, 백엔드, 테스트 문서가 달라졌다면 함께 갱신한다.
- 완료 보고에는 Plan 경로, 하네스 로그 경로, 주요 변경, 검증 결과를 포함한다.
- Plan 완료 후 커밋하려면 [Git 작업 규칙](GIT_WORKFLOW.md)과 [커밋 전 검증](PRE_COMMIT_VALIDATION.md)을 따른다.
