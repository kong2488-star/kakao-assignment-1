# 하네스 작업 로그 운영 규칙

## 목적

하네스 작업 로그는 코딩 에이전트가 승인된 Plan을 어떻게 구현하고 검증했는지 재현·감사할 수 있도록 남기는 요약 기록이다. 애플리케이션의 HTTP 요청 로그나 사용자의 활동 감사 로그와는 다르다.

로그에는 유용한 작업 증거만 남긴다. 에이전트의 내부 추론 전문, 전체 대화, 전체 터미널 출력 또는 전체 파일 내용은 저장하지 않는다.

## 생성과 파일 이름

- 코드, 설정, 테스트, 의존성 또는 코드 생성 결과를 변경하는 작업마다 로그가 필요하다.
- 승인된 Plan이 `In Progress`로 바뀔 때 `logs/TEMPLATE.md`를 복사해 로그를 생성한다.
- Plan과 로그는 동일한 세 자리 번호와 영문 kebab-case 제목을 사용한다.

```text
plans/001-project-setup.md
logs/001-project-setup.md
```

- 하나의 Plan에는 하나의 로그 파일만 사용한다.
- 완료된 로그는 이동하거나 삭제하지 않고 `logs/`에 보존한다.
- Markdown 문서 또는 동작에 영향을 주지 않는 주석만 변경하는 Plan 예외 작업에는 로그가 필요하지 않다.

## 갱신 시점

로그는 작업 종료 후 기억에 의존해 일괄 작성하지 않는다. 다음 사건이 발생하면 작업 중에 수시로 갱신한다.

- 구현 방향에 영향을 주는 결정
- 승인 범위 안에서 선택한 중요한 구현 세부사항
- 테스트 실패, 반복 오류, 구현 방향 변경 또는 환경 제약
- 의미 있는 코드·설정·테스트 변경 완료
- lint, typecheck, build, 테스트 또는 수동 검증 실행

완료 직전에는 작업 요약, 변경 파일, 검증 결과와 미해결 사항을 실제 상태에 맞게 확정한다.

## 필수 구성

모든 하네스 로그는 다음 항목을 포함한다.

- Plan 번호와 Plan 경로
- 로그 상태, 시작일, 완료일
- 작업 요약
- 주요 결정과 근거
- 변경 파일
- 의미 있는 명령과 결과 요약
- 의미 있는 실패, 원인과 해결 과정
- 검증 결과
- 미해결 사항과 후속 작업

해당 항목이 없으면 섹션을 삭제하지 않고 `없음`이라고 기록한다.

## 기록할 내용

### 주요 결정

- 승인된 Plan을 실제 코드 구조에 적용하면서 선택한 중요한 구현 방법
- 대안을 선택하지 않은 이유
- 재승인이 필요하지 않은 범위 안의 설계 세부 조정

결정의 결과와 짧은 근거만 기록하며 내부 추론 전문은 기록하지 않는다.

### 명령과 도구

다음과 같이 결과 확인에 가치가 있는 작업만 기록한다.

- 패키지 설치 또는 코드 생성
- 데이터베이스 초기화나 마이그레이션
- lint, format check, typecheck, build, pytest
- 오류 재현이나 원인 확인에 사용한 핵심 명령
- 실행 환경의 제약을 확인한 명령

단순 파일 읽기, 반복 검색, 디렉터리 조회, 성공한 사소한 명령은 기록하지 않는다. 전체 출력을 복사하지 않고 명령, 종료 상태와 핵심 결과만 요약한다.

### 실패와 해결

다음처럼 후속 작업이나 재현에 가치가 있는 실패만 기록한다.

- 테스트 또는 빌드 실패
- 같은 원인의 반복 오류
- 구현 방향을 바꾸게 만든 문제
- 권한, 네트워크, 런타임 또는 도구 제약
- 완료 시점에도 남아 있는 문제

오타, 즉시 수정된 경로 실수처럼 학습 가치가 없는 일시적 실패는 생략할 수 있다.

## 기록하지 않을 내용

다음 정보는 로그에 원문으로 저장하지 않는다.

- API 키, 토큰, 쿠키, 비밀번호와 인증 헤더
- `.env`와 환경변수의 실제 값
- 개인정보 또는 사용자 입력 원문
- 요청·응답 본문과 Todo 제목
- 전체 데이터베이스 내용
- 에이전트의 내부 추론 전문
- 전체 터미널 출력 또는 전체 파일 내용

필요한 맥락을 유지해야 한다면 민감한 값을 `[REDACTED]`로 대체한다. 비밀정보가 도구 출력에 나타났더라도 로그로 옮기지 않는다.

## 상태와 완료 조건

로그 상태는 아래 순서로 관리한다.

```text
In Progress → Completed
```

- 생성 시 상태는 `In Progress`이고 시작일을 기록한다.
- Plan이 재승인을 위해 `Draft`로 돌아가면 기존 로그에 중단 사유를 기록하고 추가 구현 기록을 멈춘다.
- 재승인 후 같은 Plan 번호와 로그 파일을 계속 사용한다.
- 구현과 검증이 끝나면 요약과 모든 필수 섹션을 갱신하고 로그 상태와 완료일을 기록한다.
- 하네스 로그가 완성되지 않으면 대응 Plan을 `Completed`로 변경할 수 없다.
- Plan, 로그, 실제 변경 파일과 검증 결과가 서로 일치해야 한다.
- 커밋 전 `PreStage`와 `Staged` 결과, staged 파일 목록과 제안 커밋 메시지를 기록한다.

## 애플리케이션 로그와의 구분

이 문서는 코딩 에이전트의 작업 기록만 다룬다. Next.js 또는 FastAPI의 HTTP 로그, 운영 모니터링, 파일 회전, 로그 보존 기간, 외부 수집 서비스는 별도 요구사항과 Plan이 있을 때 설계한다.
